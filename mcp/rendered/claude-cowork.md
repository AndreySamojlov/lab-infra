# Claude Cowork / Claude Desktop — MCP connectors

Generated by `scripts/render-mcp-configs.py` from `mcp/servers.json`.
Do not edit this file manually — your changes will be overwritten.

Client: **claude-cowork** — Claude running inside the Cowork desktop app.

Cowork on Windows is MSIX-packaged, so its config lives in the
virtualized AppData path, not the usual `%APPDATA%\Claude\`:

```
%LOCALAPPDATA%\Packages\Claude_pzs8sxrjxfjjc\LocalCache\Roaming\Claude\claude_desktop_config.json
```

Merge the `mcpServers` block below into the existing file, keeping
any `preferences` block intact. Claude Desktop and Cowork share
this config; Cowork additionally bridges these servers into its
sandboxed VM as `type: "sdk"` entries visible in the `/mcp` dialog.

## Prerequisites

- Node.js >= 18 on the host (`node -v`). Install via
  `winget install OpenJS.NodeJS.LTS` or from nodejs.org.
- PowerShell ExecutionPolicy must allow npm/npx shims:
  `Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned`.
- For each server with bearer auth, set the referenced token as a
  Windows **user** environment variable before launching Cowork:
  `setx <VAR> "<value>"` (new shells only) or
  `[System.Environment]::SetEnvironmentVariable('<VAR>', '<value>', 'User')`.

After editing the config and setting env vars, fully quit Cowork
through the tray icon (not just close the window) and relaunch.

## n8n-mcp

Community n8n-mcp server (rich builder layer on top of the n8n API).

- **URL**: `https://n8n-mcp.samandrey.work/mcp`
- **Transport**: `http` via local stdio bridge (`mcp-remote`)
- **Required Windows user env var**: `N8N_MCP_AUTH_TOKEN` (value from `.env` variable `N8N_MCP_AUTH_TOKEN` on the platform VM)

**`claude_desktop_config.json` block** (merge into `mcpServers`):

```json
"n8n-mcp": {
  "command": "npx",
  "args": [
    "-y",
    "mcp-remote",
    "https://n8n-mcp.samandrey.work/mcp",
    "--header",
    "Authorization:${AUTH_HEADER}"
  ],
  "env": {
    "AUTH_HEADER": "Bearer ${N8N_MCP_AUTH_TOKEN}"
  }
}
```

## Verification

After relaunching Cowork, open the `/mcp` dialog. The UI indicator
may briefly show `failed` during the first `npx` cold start — that
is a known UI/runtime race with `mcp-remote`. The real check is
whether the server's tools actually respond: run a minimal probe in
chat (e.g. ask Claude to list workflows). Detailed bridge logs:
`%USERPROFILE%\.mcp-auth\*_debug.log`.
